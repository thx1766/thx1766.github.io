CS 198:431 Software Engineering Project

The course project in this class are run along the lines originally laid out by Elaine Kant at CMU in the early 80's. (See ACM Software Engieering Notes, 1981). Basically, these involve teams of 4-5 people, but I tend to require everyone to work on the same general problem in order to both make grading fairer, and to allow me to take parts of lectures to discuss problems I encountered with the project documents as they are being submitted, which should then be of interest to everyone.

The document that describes the general aspects of the project is available as a 16 page pdf file here, or as an 8 page (2 pages to 1) reduction here. Please read it immediately!

Working in teams

The following list is a very short summary of suggestions, and is reproduced from Professor Steve Easterbrook's notes on Software Engineering projects:

  • The first thing to do is to find some times when you can all meet as a team (we ask you to decide on the times for these on the team information sheet).
  • If you decide to drop the course, please notify your partners and the tutor/professor as early as possible.
  • Arrange for at least one full team meeting per week, outside the timetabled sessions, and ensure that all members are available at the designated time. Be sensitive to each other's constraints in scheduling these - especially as some students cannot be on campus every day of the week.
  • Produce a written agenda for each meeting, and designate a chairperson and a secretary. The secretary should take minutes, and type them up for circulation by the next meeting. Rotate these roles during the term.
  • Do not take important decisions at meetings if the whole team is not present. By all means discuss the issues, but save the decision until you are all there.
  • Try not to resort to voting too often. If you find you have to vote on every decision, then its time to rethink how you arrive at consensus. In particular, if the same person or persons lose every vote, then your team is in danger of fragmenting. Bear in mind that to be a productive team, you need to learn to listen to one another.
  • Try not to get personally attached to ideas. Use techniques such as brainstorming, in which the team generates as many ideas a possible without any reflection on whether the ideas coming out are sensible or not. At the end of the exercise, all ideas are owned by the team as a whole: there should be no attempt to remember who said what.
  • You may want to assign overall responsibility for the various tasks involved in each assignment to a particular team member. Alternatively, you may want to assign specific roles to team members, such as:
    • Overall responsibility for putting together the write-up for each assignment;
    • Overall responsibility for asigning tasks and checking everyone's progress;
    • Overall responsibility for managing risk;
    • Overall responsibility for each change request (assignments 2 and 4);
    • Chief reviewer for code checked back in to the repository
    • etc.

These are general and much more extensive notes on working in groups on software engineering projects at university. In my opinion they are outstanding -- the result of many years of experience by the author as TA and instructor for project courses. The material is presented with the kind permission of Professor Stephanie Ludi. I strongly recommend reading Chapters 2,3,5,6,7 and Part III, right at the beginning of the course.

Note that in this course, I normally refrain from taking action if there are problems within a team, because problem resolution is a key skill that I hope you all learn. The peer evaluation form that you will fill out for everyone on your team is one place where you will be able to inform me of difficulties/problems, and is one "stick" that you can use (sparingly) to deal with difficulties.

Project deliverables (and what we look for while grading):
The following is a list of deliverables, described in the handout which you have received, together with due dates. It is your responsibility to read the above handout and to prepare these documents on time. Except for code, which can be ASCii, all deliverables must be PDF files and should be submitted via Sakai by 11:55pm of the dates indicated below.

Please make sure that every submission identifies the team, the team-members contributing, the editor, and date; and it must have an Acknowledgements section indicating who contributed what part, ( This is useful for knowing to whom to direct questions, ) as well as outside resources used. Please be sure to prepare a small such document even for submissions which are tar files for the rapid prototype or the final system code.
  • Team list, including contact email and phone for each member, and the meeting time agreed on. (Week 1) (Sept 4)
  • Storyboards/Prototype Implementation (Week 3) (Sept. 15) "walkthrough/demo" in recitation
    • User Interface
    • Functionality
    • Examples worked out for all normal case uses. (no error checking,...)
    • Some code showing your understanding of how to implement the so-called Model-Observer pattern using the Java SWING library (technology you will likely be using).
  • Requirements Document (Week 4) (Sept 25)
    • Use-cases and scenarios (supported by specific examples) rather than verbose descriptions
    • Thoughtful subsetting into "kernel", "reasonable", "advanced",... categories. (All projects find themselves running out of time at the end of the term, and this subsetting will also be used for design purposes.)
    • Abnormal/error conditions as well as normal case handling.
    • Precise descriptions whenever possible (Use UML notations, grammars, automata, regular expressions when appropriate.)
    • Non-functional requirments.
  • Architectural Design Document (Week 5/6) (Oct 9th)
    • Good modularization, which supports information hiding/abstraction/implementation secrets, allowing easier modifications later on.
    • Modularization that helps the "subsetting" suggested in the requirements.
    • Systematic specification of each module (including imports, exports) and procedure (requires, effect, and error signals).
    • Appropriate use of iterators.
        • If you are interested, a former TA (Tanvir) has researched custom tags in javadoc, and this tar file contains i) some Java files needed to use tags like @requires, @effect, @modifies (others, like @param, @returns, @throws are built in); ii) an example of a search program annotated as I discussed in class; iii) a short html file explaining custom tags.
  • Design Walkthrough (Scheduled with each group separately Oct. 10-12)
  • Detailed Design Document (week 8) (Oct. 30)
    • Fixing problems revealed in walkthrough. (It is perfectly reasonable to change the architecture of the software, to improve it.)
    • Continue making description more precise, using assertions (rep. invariants for data structures, assumes/postcondition/returns for procedures/methods)
    • Code reuse through libraries.
  • Coding (Weeks 10-14)
    • Clean, well-documented code
  • Test Plan Document (Week 11)(Nov. 20)
    • Each student must provide good black-box and clear-box test data for at least one interesting class which is going to be tested by him/her.
    • Precise integration plan with dates.
  • Preliminary system demonstration (at least kernel system running) (Dec. 6th)
  • Final System Delivery/Freeze (last day of classes (Dec. 13th)
  • System Demonstration (to be scheduled during reading days)
    • Make sure you practice a 'script' that shows off everything you have done, without bugs showing up.
    • Afterwards we will 'take control' to try to wreck things.

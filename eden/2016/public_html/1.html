1. 1.Intro 

    1. 1.What is an Operating system? 

      1. 1.Software layer between hardware and application programs/users which provides a virtual machine interface 

      2. 2.A resource manager that allows programs/users to share hardware resources 

        1. 1.Dishes out resources in a fair and efficient manner 

      3. 3.Set of utilities that simplify development 

    2. 2.Why use an OS? 

      1. 1.Benefits for application writers 

        1. 1.Easier to write programs, Portability 

      2. 2.Benefits for users 

        1. 1.Easier to use computers, protects them from themselves/others 

    3. 3.Mechanism and Policy 

      1. 1.Mechanisms: data structures and operations that implement an abstraction (file system buffer cache, etc) 

      2. 2.Policies: the procedures that guide the selection of a certain course of action from among alternatives (replacement policy of the buffer cache, etc) 

    4. 4.Processes 

      1. 1.System abstraction: illusion of being the only job in the system 

      2. 2.Mechanism 

        1. 1.Creation, destruction, suspension, context switch, signaling, IPC, etc 

      3. 3.Policy 

        1. 1.Minor policy 

          1. 1.Who can create/destroy/suspend processes? 

          2. 2.How many active processes can each user have? 

        2. 2.Major policy 

          1. 1.How to share system resources between processes? 

          2. 2.Typically broken into a number of orthogonal policies for individual resources, such as CPU, memory, and disk 

    5. 5.Processor abstraction: threads 

      1. 1.A thread is a processor abstraction. Gives the illusion of having 1 processor per execution context. 

      2. 2.Mechanisms 

        1. 1.Creation, destruction, suspension, context switch, signaling, synchronizing etc 

      3. 3.Policy 

        1. 1.How to share CPU between threads from same/different processes? 

        2. 2.How can multiple threads synchronize with each other? 

        3. 3.How to control inter-thread interactions? 

          1. 1.Can a thread kill other threads just because it wants? 

    6. 6.Memory Abstraction: Virtual Memory 

      1. 1.Virtual memory is a memory abstraction, gives the illusion of large contiguous memory, often more memory than what's physically available 

      2. 2.Mechanism  

        1. 1.Maps virtual memory to physical memory, handles page-faults etc 

      3. 3.Policy 

        1. 1.How do we multiplex a virtual memory that is larger than the physical memory onto what's available? 

        2. 2.How should physical memory be allocated to competing processes? 

        3. 3.How to control the sharing of a piece of physical memory between multiple processes? 

    7. 7.Storage Abstraction: File System 

      1. 1.A file system is a storage abstraction: gives the illusion of a structured storage space 

      2. 2.Mechanism 

        1. 1.File creation, deletion, read, write, file-block-to-disk-block mapping, buffer cache, etc 

      3. 3.Policy 

        1. 1.Sharing vs protection? 

        2. 2.Which block to allocate? 

        3. 3.File system buffer cache management? 

    8. 8.Communication Abstraction: Messaging 

      1. 1.Message passing is a communication abstraction: illusion of a reliable and ordered transport 

      2. 2.Mechanism  

        1. 1.Send, receive, buffering, retransmission, etc 

      3. 3.Policy 

        1. 1.Congestion control and routing 

        2. 2.Multiplexing multiple connections onto a single network interface 

    9. 9.Character and Block Devices 

      1. 1.The device interface gives the illusion that devices support the same API: character stream and block access 

      2. 2.Mechanisms 

        1. 1.Open, close, read, write, IO Control, etc 

        2. 2.Buffering 

      3. 3.Policies 

        1. 1.Protection 

        2. 2.Sharing? 

        3. 3.Scheduling? 

    10. 10.What are the major issues in OS design? 

      1. 1.Programming the API: what should the virtual machine look like? 

      2. 2.Resource Management: how should the hardware resources be multiplexed among multiple users? 

      3. 3.Sharing: how should resources be shared among multiple users? 

      4. 4.Protection: how to protect users from each other? How to protect programs from each other? How to protect the OS from applications and users? 

      5. 5.Communication: how can applications exchange information? 

      6. 6.Structure: how do we organize the OS? 

      7. 7.Concurrency: how do we deal with the concurrency that is inherent in Oses 

      8. 8.Performance: how do we make it all run fast? 

      9. 9.Reliability: how do we keep the OS from crashing? 

      10. 10.Persistence: how can we make the data last beyond program execution? 

      11. 11.Accounting: how do we keep track of resource usage? 

      12. 12.Distribution: how do we make it easier to use multiple computers in conjunction? 

      13. 13.Scaling: how do we keep the OS efficient and reliable as the offered load increases? (more users, processes, processors) 

    11. 11.Spooling 

      1. 1.CPUs were much faster than card readers, then disks were faster than those, so what do we do? 

        1. 1.Read 1 job from disk, run job 1 while reading job 2 from cards to disk, save output from 1 to disk, print output of job 1 while running job 2 while reading job 3 from cards to disk etc.  

        2. 2.Spooling: Simultaneous Peripheral Operation On Line 

      2. 2.Can use multiple card readers and printers to keep up with CPU if needed 

      3. 3.Improves both system throughput and response time 

    12. 12.Multiprogramming 

      1. 1.CPUS were still idle whenever the program needed to interact with peripheral device 

      2. 2.Multiprogramming batch systems invented 

        1. 1.Load multiple programs onto disk at the same time 

        2. 2.Switch from one job to another when first performs an IO operation  

        3. 3.Overlap IO of one job with computation of another job 

        4. 4.Peripherals have to be asynchronous 

        5. 5.Have to know when IO operation is done (interrupt vs polling) 

      3. 3.Increase system throughput, possibly at the expense of response time 

        1. 1.When is this better for response time? When is it worse? 

    13. 13.Time Sharing 

      1. 1.Connect multiple terminals to same machine, multiplex machine between users, machine has to be fast enough to give illusion that each user has own machine, Multics was first large time-sharing system (1960's) 

    14. 14.Real time OS 

      1. 1.What happens when applications have deadlines by when they have to complete certain stuff? 

        1. 1.Hard real time system 

          1. 1.Medical imaging, industrial control, etc 

          2. 2.Catastrophic failure if system misses a deadline 

            1. 1.What happens if collision avoidance software on google car stops working? 

          3. 3.Challenge lies in how to meet deadlines with minimal resource waste 

        2. 2.Soft real time system 

          1. 1.Multimedia applications 

            1. 1.Annoying but not catastrophic if deadlines missed 

            2. 2.Challenge lies in how to meet most deadlines with minimal resource waste 

            3. 3.Also lies in how to load-shed if become overloaded 

    15. 15.Distributed OS 

      1. 1.Clustering 

        1. 1.Use multiple small machines to handle large services 

          1. 1.Cheaper than using one large machine 

          2. 2.Better potential for reliability, incremental scalability, and absolute scalability 

      2. 2.Wide-area distributed systems 

        1. 1.Allow use of geographically distributed resources 

          1. 1.Local use of PC to access internet services like google, amazon (don't need to carry needed info with us) 

      3. 3.Need OS support for communication and sharing of distributed resources 

        1. 1.Network file systems for example (shared drives) 

      4. 4.Want performance (speedup isn't a metric of interest here), high reliability, and use of diverse resources 

    16. 16.Embedded OS 

      1. 1.Pervasive computing 

        1. 1.Cellphones, cars, etc 

        2. 2.Android wear, etc 

      2. 2.Characteristics 

        1. 1.Constrained resources (slow cpu, low memory, etc) 

        2. 2.How can we execute more powerful programs if our hardware is similar to old hardware? 

          1. 1.Use lots of them, augment with services running on powerful machines 

      3. 3.OS support for power management, mobility, resource delivery, etc 

    17. 17.Virtual Machines and Hypervisors 

      1. 1.Popular in the 60's and 70's, vanished in 80's and 90's 

      2. 2.Idea: partition a physical machine into a number of virtual ones 

        1. 1.Each vm behaves as a separate computer 

        2. 2.can support heterogeneous operating systems (guest OS's) 

        3. 3.Provides performance isolation and fault isolation 

        4. 4.facilitates virtual machine migration 

        5. 5.facilitates server consolidation
